# Worktree Parity Audit

Confirm that running the project's back-pressure suite (tests, linters, typecheckers, codegen, formatters) inside a fresh Sortie worktree produces the same result as in the root checkout. Identify gaps, map them to Sortie config knobs, and produce a findings report.

## Why this matters

A worktree shares the same `.git` directory but starts with only the tracked working tree. Everything `.gitignore`d is absent on creation: dependency caches, env files, build outputs, secrets, local DBs, generated code. If any of those are required by the back-pressure suite, agents will fail inside the worktree in ways that don't reproduce at root — and the failure is usually invisible to the user until a task stalls.

The fix is mechanical: enumerate the gaps, decide for each whether to **recreate** it (`worktree-setup-commands`) or **share** it from the root (`worktree-sync-paths`).

## Audit Procedure

Run from the project root. Work through the steps in order; later steps depend on earlier ones.

### Step 1: Detect project type(s)

Look for marker files. Projects may be polyglot (e.g., Rails + Node assets):

| Marker | Stack |
|---|---|
| `package.json` | Node / TypeScript |
| `go.mod` | Go |
| `Cargo.toml` | Rust |
| `Gemfile` | Ruby / Rails |
| `pyproject.toml`, `requirements.txt`, `Pipfile`, `setup.py` | Python |
| `mix.exs` | Elixir / Phoenix |
| `build.gradle`, `pom.xml` | JVM (Gradle / Maven) |
| `composer.json` | PHP |
| `Package.swift` | Swift |

For each detected stack, load the matching section of `references/project-type-recipes.md` for typical gitignored items, standard install commands, and known gotchas.

### Step 2: Find the golden command

This is the single most important step. CI is the authoritative recipe — mirror what it actually runs, not a guess. Search in this order:

1. **GitHub Actions**: `.github/workflows/*.{yml,yaml}` — look for `jobs.<j>.steps` with `run:` blocks. The job named `test`, `ci`, `lint`, or `check` is usually authoritative.
2. **Other CI**: `.gitlab-ci.yml`, `.circleci/config.yml`, `azure-pipelines.yml`, `buildkite.yml`.
3. **Build scripts**: `Makefile` (targets `test`, `check`, `ci`, `verify`), `Justfile`, `Taskfile.yml`.
4. **Package script tables**: `package.json#scripts.{test,ci,check,lint,typecheck}`, `pyproject.toml#tool.poetry.scripts`.
5. **Convention scripts**: `bin/ci`, `bin/test`, `script/test`, `bin/setup`.

Extract three things:
- The **install / setup commands** (dep fetch, codegen, asset build, DB setup) in order.
- The **verify command(s)** — the actual test/lint/typecheck invocations.
- Any **env vars** the CI job sets explicitly (`env:` blocks in workflow files).

If no CI exists, ask the user for the canonical local recipe — do not invent one.

### Step 3: Classify `.gitignore` entries

Read every `.gitignore` (root + nested). For each rule, place it in one of these buckets:

| Category | Examples | Disposition |
|---|---|---|
| Build artifacts (regenerated by golden command) | `dist/`, `.next/`, `__pycache__/`, `target/debug/`, `_build/` | **Skip** — the golden command rebuilds. |
| Dependency caches (slow to recreate) | `node_modules/`, `venv/`, `vendor/bundle/`, `deps/` | **Recreate** via `worktree-setup-commands` OR **share** via `worktree-sync-paths.link`. See trade-offs below. |
| Env files | `.env`, `.env.local`, `.env.test` | **Share** via `worktree-sync-paths.link`. Use `copy:` if per-worktree overrides are needed. |
| Secrets / keys | `*.pem`, `config/master.key`, service-account JSON | **Share** via `worktree-sync-paths.link`. Flag to user that the file must exist at root. |
| Local databases | `db/*.sqlite3`, `*.duckdb` | **Init per-worktree** via `worktree-setup-commands` (migrate + seed). Sharing causes lock contention. |
| Logs / tmp | `log/`, `tmp/` | **Skip** — per-worktree by design. |
| Editor / OS noise | `.DS_Store`, `.vscode/`, `*.swp` | **Skip**. |

#### Recreate vs. share — trade-offs

- **Recreate (`worktree-setup-commands`)**: clean isolation, deterministic. Slower per-worktree but right default when install is fast (`go mod download`, `pnpm install --frozen-lockfile` on a warm cache, `uv sync`).
- **Share via `link:`**: instant. But Sortie uses **hard-links, not symlinks** — implications:
  - Hard-links cannot cross filesystems. If `.sortie/worktrees/` lives on a different volume from the main checkout, `link:` fails — fall back to `copy:` or recreate.
  - Edits propagate (shared inode). Right for env files and read-only deps; wrong for anything stateful per task.
- **Share via `copy:`**: independent per-worktree, no inode sharing. Fast but uses more disk. Use for env files where each task may mutate values.

### Step 4: Audit env var usage

Grep code for env var reads and compare against `.env.example` / `.env.sample`:

```bash
# JS/TS
rg "process\.env\.([A-Z_][A-Z0-9_]*)" -o -r '$1' --no-heading | sort -u
# Python
rg "os\.(getenv|environ\[)" --no-heading
# Ruby
rg "ENV\[['\"]([A-Z_][A-Z0-9_]*)['\"]\]" -o -r '$1' --no-heading | sort -u
# Go
rg "os\.Getenv\(['\"]([A-Z_][A-Z0-9_]*)['\"]\)" -o -r '$1' --no-heading | sort -u
# Rust
rg "(std::)?env::var\(['\"]([A-Z_][A-Z0-9_]*)['\"]\)" -o -r '$2' --no-heading | sort -u
```

Vars present in code but missing from `.env.example` mean the user must supply them — flag explicitly in findings. The audit cannot guess secret values.

### Step 5: Hidden setup steps from CI

Beyond `<install> && <run tests>`, CI often does work that breaks in a fresh worktree if skipped:

- **DB setup**: `bin/rails db:prepare`, `pnpm prisma migrate deploy`, `alembic upgrade head`
- **Codegen**: `go generate ./...`, `cargo build --workspace`, `mix compile`, `pnpm run codegen`
- **Asset compilation**: `bin/rails assets:precompile`, `pnpm build`
- **Test data seeding**: fixture loads, factory setup

Each becomes a candidate `worktree-setup-commands` entry. **Order matters** — install deps before codegen before DB setup.

### Step 6: Path-relative and tooling assumptions

Quick sanity checks:

- `rg '\.\./\.\.' -tjs -tts -tgo -trust -tpy -truby` — `../..` patterns. Worktrees live at `.sortie/worktrees/<task>/`, so paths internal to the worktree behave identically; paths escaping the working tree to a sibling project will silently misbehave.
- Tool-version managers: `.nvmrc`, `.tool-versions`, `.python-version`, `rust-toolchain.toml` — usually fine (they configure tools globally), but flag if the project uses per-project tool installs (e.g., `mise` shims tied to the working tree).
- Git ops: `git tag`, `git log`, `git rev-parse` mostly work in a worktree because `.git` is shared, but flag code that assumes the working tree is a top-level clone (rare).

### Step 7: Verify parity (optional but recommended)

The only true confirmation is running the golden command in a fresh worktree:

```bash
# From repo root
git worktree add /tmp/parity-test HEAD
cd /tmp/parity-test
# Run the install/setup steps from Step 5 in order, then the verify command
# (and any required env-file copies / symlinks that mirror the planned config).
```

If it passes here, it will pass in Sortie's worktrees (modulo the hard-link cross-FS caveat). If it fails, the audit missed something — iterate.

Clean up: `git worktree remove /tmp/parity-test`.

## Output Format

Produce a findings report inline (no file write unless the user asks). Use this template:

```markdown
# Worktree Parity Audit — <project name>

## Detected stack
- <stack> (marker: `<file>`)
- <stack> (marker: `<file>`)

## Golden command
Source: `<file>:<line>`

\```
<setup commands in order>
<verify command>
\```

## Gaps

### Dependencies
- `<path>` — gitignored, required by <which step>. Recommend: `<knob>` → `<value>`.

### Env / secrets
- `<file>` — gitignored, contains <vars>. Recommend: `worktree-sync-paths.link: [<file>]`. Note: user must ensure this file exists at the root checkout.

### Required env vars not in `.env.example`
- `<VAR>` (referenced at `<file>:<line>`) — flag for user.

### Setup commands beyond install
- `<command>` — source: `<file>:<line>`.

## Recommended `.sortie.yml` snippet

\```yaml
worktree-sync-paths:
  link:
    - <files>

worktree-setup-commands:
  - <install>
  - <codegen>
  - <db setup>
\```

## Caveats
- <hard-link cross-FS note, if relevant>
- <inode-sharing note for env files, if relevant>
- <any per-worktree state concerns>

## Verification
- [ ] Ran golden command in a `git worktree add` scratch worktree with the recommended setup. Exit 0.

## Next step
Hand this snippet to `sortie-configurer` to integrate into `.sortie.yml`, or paste it manually.
```

## Mapping gaps → Sortie knobs

| Gap | Knob |
|---|---|
| Dep dir, fast install | `worktree-setup-commands: [<install cmd>]` |
| Dep dir, slow install, same-FS | `worktree-sync-paths.link: [<dir>]` |
| Dep dir, cross-FS | `worktree-sync-paths.copy: [<dir>]` or install command |
| Env file, single shared value | `worktree-sync-paths.link: [.env*]` |
| Env file, per-worktree override | `worktree-sync-paths.copy: [.env]` then mutate via setup command |
| Codegen / DB setup | `worktree-setup-commands` (ordered) |
| Tool-version file (`.nvmrc` etc.) | Committed, no action |
| Secret file (gitignored) | `worktree-sync-paths.link: [<file>]` (root must have it) |

## Hand-off

This skill stops at findings + snippet. Writing the snippet into `.sortie.yml` (with correct kebab/snake_case keys, validation, etc.) is `sortie-configurer`'s job. If the user wants the config edit applied immediately, state so explicitly and let `sortie-configurer` take over.
